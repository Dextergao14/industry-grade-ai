## 2.1 全景速览:Transformer 到底解决了什么 🔹

### 2.1.1 一句话版本

Transformer 是一个把"序列中的每个词"变成"充分吸收了上下文信息的词"的机器,而它吸收上下文的方式,是让每个词直接和其他所有词"对话"——这个对话机制,就是注意力。

### 2.1.2 它取代了谁,凭什么

在 2017 年之前,处理序列的主流是 RNN(以及它的改良版 LSTM)。RNN 读句子像**传话游戏**:第 1 个词把信息告诉第 2 个词,第 2 个词消化后再告诉第 3 个……传到第 500 个词时,开头的信息早就走样了;而且必须一个一个传,GPU 上一万个计算单元只能干等。

Transformer 把传话游戏改成了**圆桌会议**:所有词同时入座,任何两个词之间都是一步直达,想听谁的直接听。这带来两个决定性优势:

1. **长程依赖**:第 500 个词和第 1 个词的距离不再是 499 步,而是 1 步。信息不衰减。
2. **并行**:一句话里所有位置的计算可以同时做,完美匹配 GPU 的并行架构。

面试一句话答法:"RNN 的问题是串行计算和长程信息衰减,Transformer 用注意力把任意两个位置的交互距离降到 O(1),同时让整个序列可以并行计算。"

### 2.1.3 现代 LLM 的骨架:decoder-only

原始论文里的 Transformer 是 encoder-decoder(为翻译设计)。今天你面试会遇到的 GPT、LLaMA、Qwen、DeepSeek,几乎全是 **decoder-only**:只保留"根据前文预测下一个词"这一个任务。它的数据流只有五步:

```
文本 → ① tokenizer 切成 token id
     → ② embedding 查表,每个 token 变成一个 d_model 维向量
     → ③ N 层 Transformer Block(每层 = 注意力 + FFN,配残差和归一化)
     → ④ 最后一层输出过一个线性层,映射到词表大小
     → ⑤ softmax 得到"下一个词"的概率分布
```

各组件分工一张表(细节在标注的小节展开):

| 组件 | 干什么 | 一句话直觉 | 详见 |
|---|---|---|---|
| Embedding | token id → 向量 | 给每个词发一张可学习的"身份卡" | 3.3 节(tokenizer) |
| **注意力** | 让 token 之间交换信息 | 开会,互相听取发言 | **本章 2.2–2.3** |
| FFN | 对每个 token 独立做非线性变换 | 开完会各自回家消化、查资料 | 2.5 节 |
| 残差连接 | 每层的输出 = 输入 + 本层增量 | 只写"修改批注",不推翻重写 | 2.4 节 |
| 归一化 | 稳定每层的数值范围 | 每层开会前先把音量调齐 | 2.4 节 |
| 位置编码 | 告诉模型词的先后顺序 | 给圆桌会议发座位号 | 2.4 节(RoPE) |

一个常被追问的分工直觉:**注意力负责"token 之间"的信息流动,FFN 负责"token 内部"的加工与知识存储**。开会(attention)交换信息,回家(FFN)各自消化——整个 Transformer 就是 N 轮"开会—消化"的循环。

速览到此为止。下面进入正题。

## 2.2 注意力机制 🔸

### 2.2.1 从查字典说起:硬查表 → 软查表

先忘掉神经网络,看一行谁都懂的代码:

```python
d = {"苹果": "水果", "汽车": "交通工具"}
d["苹果"]   # → "水果"
```

字典查询有三个角色:你手里的**查询词**(query)、字典里的**键**(key)、键对应的**值**(value)。查询的规则是:query 和哪个 key **完全相等**,就返回哪个 value。

现在提一个要求:我想查 `"梨"`。硬字典直接 KeyError。但直觉上,"梨"和"苹果"很像,理想的返回结果应该**接近**"水果"。于是我们把规则从"完全相等"放宽为"按相似度加权":

> 拿 query 和**每一个** key 算相似度 → 相似度归一化成权重(加起来等于 1)→ 用权重对**所有** value 加权平均。

这就是**软查表**。"梨"和"苹果"相似度 0.9、和"汽车"相似度 0.1,返回 0.9×"水果" + 0.1×"交通工具"——一个"基本是水果"的答案。没有 KeyError,处处可导,能进神经网络训练。

**注意力机制 = 可学习的软查表。** 公式还没出场,但机制已经讲完了,剩下的全是细节:相似度怎么算、query/key/value 从哪来。

### 2.2.2 核心比方:开会举手发言

把一句话想成一场圆桌会议,每个 token 是一位参会者。每位参会者带三样东西入场:

- **Q(query)**:"我想听什么"——我当前缺什么信息;
- **K(key)**:"我能讲什么"——我这里有什么信息,相当于我举的牌子;
- **V(value)**:"我实际讲的内容"——真正被传递的信息。

会议规则:每位参会者拿自己的 Q,去看**所有人**举的 K 牌。牌子和需求越匹配,就越认真听那个人发言(V),最后把听到的所有发言按认真程度加权汇总,更新自己的认知。

举个具体句子:"银行旁边的河很清澈"。"银行"这个词是多义的(bank:金融机构/河岸)。它的 Q 大致是"谁能帮我消歧?";"河"举的 K 牌恰好匹配,于是"银行"重点听取"河"的 V,更新后的"银行"向量就带上了"河岸"的含义。**这就是"上下文相关的词表示"的来源——attention 输出的每个词向量,都是被上下文重新定义过的。**

三个细节,都是面试追问点:

1. **为什么 K 和 V 要分开?** 你举的牌子和你实际讲的内容本来就是两回事——牌子是为了"被匹配到"而优化的(广告),内容是为了"有用"而优化的(干货)。让两者共用一个向量会互相牵制;分开各学各的,表达力更强。
2. **Q、K、V 从哪来?** 都是同一个输入向量 x 乘以三个**可学习**矩阵:Q = xW_q,K = xW_k,V = xW_v。同一个人,分别戴上"提问者""被查者""发言者"三顶帽子。模型训练的过程,就是在学怎么提问、怎么举牌、怎么发言。
3. **自己听自己吗?** 听。每个 token 的 Q 也会和自己的 K 算相似度,通常权重还不小——"我已有的信息本身也重要"。

### 2.2.3 把比方写成公式

上面的会议规则,逐步翻译成数学。设一句话有 n 个 token,每个 token 已是 d 维向量,堆成矩阵 X(形状 n×d)。

**第一步,配三顶帽子:**

$$Q = XW_q,\quad K = XW_k,\quad V = XW_v$$

**第二步,算匹配度。** 用点积衡量 Q 和 K 的相似度——方向越一致点积越大。所有 token 两两之间一次算完,就是一个矩阵乘法:

$$S = QK^\top \quad (n \times n)$$

S[i][j] 的含义:第 i 个 token 对第 j 个 token 的原始关注度。这个 n×n 矩阵是整个 Transformer 的灵魂,也是它 O(n²) 复杂度的来源(2.2.6 节)。

**第三步,归一化成权重。** 对 S 的每一行做 softmax,把任意实数变成"和为 1 的正数"——即注意力权重。

**第四步,加权听取发言:**

$$\text{Attention}(Q,K,V) = \text{softmax}\!\left(\frac{QK^\top}{\sqrt{d_k}}\right)V$$

输出形状回到 n×d_v:每个 token 一个新向量,即"开完会后的认知"。

公式里凭空多了个 **除以 √d_k**——这是本章唯一一处推导,也是最高频的面试题,单独开一节。

### 2.2.4 为什么除以 √d_k:半页推导 + 一个实验 🔸

**问题**:两个向量维度 d_k 越大,点积就天然越大——不是因为更相关,纯粹是因为加的项多。大到什么程度?

**推导**(面试可要求手写,就这四行):假设 q 和 k 的每个分量独立、均值 0、方差 1(初始化时近似成立)。点积 $q \cdot k = \sum_{i=1}^{d_k} q_i k_i$。

- 每一项 $q_i k_i$:均值 0,方差 $\mathbb{E}[q_i^2 k_i^2] = \mathbb{E}[q_i^2]\,\mathbb{E}[k_i^2] = 1$;
- d_k 项独立相加:总方差 = **d_k**,标准差 = **√d_k**。

取 d_k = 64(常见 head 维度),点积的标准差是 8;d_k = 128 时是 11.3。也就是说,softmax 的输入动辄是 ±10、±20 量级的数。

**这会怎样?** softmax 对大数极其敏感:输入是 [20, 1, 0, -3] 时,输出几乎是 [1, 0, 0, 0]——**one-hot 化**。两个后果:

1. 会议变成"只听一个人发言",加权平均退化成硬查表,上下文信息大量丢失;
2. 更致命的是梯度:softmax 饱和区的梯度趋近 0,反向传播时几乎没有信号流过,**训练直接学不动**。

除以 √d_k 后,点积方差恢复为 1,softmax 工作在梯度健康的区间。**比方:√d_k 是会议室的音量旋钮——人越多(d 越大)现场越吵,旋钮按人数的平方根把音量归位,保证每个发言都还能被听见。**

眼见为实,五行代码:

```python
import torch
d = 64
q, k = torch.randn(1000, d), torch.randn(1000, d)
scores = (q * k).sum(-1)
print(scores.std())          # ≈ 8.0 ≈ sqrt(64),验证推导
print(torch.softmax(torch.tensor([8.0, 0.0, -8.0]), dim=0))
# → [0.9997, 0.0003, 0.0000]  一个未经缩放的典型行:已经 one-hot 化
```

**追问预警**:"除以 d 行不行?"——过度缩小,方差变成 1/d,分数挤在 0 附近,注意力太"平",区分度不足;√d 恰好把方差归一到 1。"训练后分布变了,还需要除吗?"——需要,缩放保证的是训练**起点**的梯度健康,起点学不动就没有后面了。

### 2.2.5 因果掩码:不许偷看答案

decoder-only 模型的训练任务是预测下一个词。如果开会时第 3 个词能听到第 5 个词的发言,那"预测第 4 个词"就是作弊——答案就坐在会议室里。

解法简单粗暴:在 softmax **之前**,把分数矩阵 S 中所有"未来位置"(j > i 的上三角)填成 **-∞**。softmax 里 e^{-∞} = 0,未来位置的权重精确为零,而其余位置的权重仍然严格归一。

```
S = [[ s11, -inf, -inf ],        权重 = [[ 1.00, 0.00, 0.00 ],
     [ s21,  s22, -inf ],   →            [ 0.4,  0.6,  0.00 ],
     [ s31,  s32,  s33 ]]                [ 0.2,  0.3,  0.5  ]]
```

两个易错点:一,必须填 -∞ 而不是 0——分数为 0 经过 softmax 仍有正权重,照样泄题;二,mask 加在 softmax 之前而不是之后——事后置零会破坏"权重和为 1"。

这个"每个 token 只能看到自己和之前"的结构,还有一个重要副产品:**生成时,已算过的 token 的 K、V 永远不变,可以缓存下来重复使用**——这就是 KV cache,推理系统的头号主角,伏笔埋在这里,第 13 章收割。

### 2.2.6 复杂度:一笔账,两个伏笔

设序列长 n、维度 d:

- 分数矩阵 QK^T:n² 次点积,每次 d 乘加 → **计算 O(n²d)**;
- 这个 n×n 矩阵本身 → **显存 O(n²)**。

n = 1k 时 n² = 100 万,无所谓;n = 128k(如今的常规上下文长度)时 n² = 160 亿——attention 矩阵一项就能吃掉几十 GB 显存。这笔账催生了本书两条重要线索:

1. **不改数学,改访存**:FlashAttention——分块计算,永远不把 n×n 矩阵完整写进显存(第 10 章);
2. **改数学**:线性注意力、稀疏注意力、SSM/Mamba(2.7 节速览)。

面试标准句:"attention 的计算复杂度是 O(n²d),显存瓶颈在 n×n 的分数矩阵;FlashAttention 不改变计算量,通过分块和 online softmax 把显存从 O(n²) 降到 O(n),同时减少 HBM 读写从而更快。"——这一句话能把面试官直接引向你准备好的第 10 章内容。

## 2.3 多头注意力(MHA)🔸

### 2.3.1 为什么一个头不够

单头注意力有个结构性瓶颈:每个 token 对全场只有**一套**注意力权重——一个 n 维的 softmax 分布。但语言里的关系是多维的,同一个词往往需要**同时**关注几路不同的信息。

看这句话:"她把咖啡递给他,因为**他**看起来很困"。"他"这个 token 至少需要:一路注意力做**指代消解**(听"他"前面出现的人物),一路捕捉**因果结构**(听"因为"),一路关注**语义关联**(听"困"和"咖啡")。一套 softmax 权重,只能把这几路需求平均成一锅粥。

**比方:一场会议,与其让一个人竖着耳朵试图同时听清所有维度,不如拆成几场并行的分组讨论**——语法组、指代组、语义组各开各的,每组用自己的 Q/K/V 各自加权,最后把各组结论拼接汇总。每个"分组讨论"就是一个**头(head)**。

事后观察也支持这个设计:对训练好的模型做可视化,确实能看到某些头专注邻近词、某些头专注指代、某些头盯着标点和句子边界。(注意分寸:这是"事后可解释的分工",不是"设计好的分工"——面试时别说反。)

### 2.3.2 维度怎么拆:参数量不变,视角变多

MHA 不是把 d 维的注意力做 h 遍(那样计算量翻 h 倍),而是**把 d_model 维切成 h 份**,每个头在 d_head = d_model / h 的低维子空间里独立做注意力:

```
d_model = 768, h = 12  →  每个头 d_head = 64

x (n×768)
 ├─ 头1: Q,K,V ∈ n×64 → attention → 输出 n×64
 ├─ 头2: 同上(独立参数,独立 softmax)
 ├─ ...
 └─ 头12: 同上
拼接 12 个 n×64 → n×768 → 乘 W_o 混合 → 输出 n×768
```

三个关键账目(都是面试题):

- **参数量**:12 个头各自的 W_q 是 768×64,合起来 768×768——和单头一模一样。**多头是"重新划分",不是"加料"**;工程上更是直接用一个 768×768 的大矩阵一次算完再 reshape 成 12 份(见代码)。
- **计算量**:同理基本不变。h 个头各算 n×n 的注意力,但每次点积从 768 维降到 64 维。
- **末尾的 W_o 干嘛的**:拼接只是把各组讨论纪要**钉在一起**,W_o 负责让各组信息**互相交换、加权融合**——没有它,12 个头到下一层还是各说各话。

**h 怎么选?** 约定俗成 d_head ≈ 64~128:头太多则每头维度太低,单头表达力不足(点积在 8 维空间里几乎区分不了什么);头太少则回到"一锅粥"问题。实践中这是个相当平坦的超参,面试答出"head_dim 通常固定在 64~128,h 随 d_model 等比增长"即可。

### 2.3.3 💻 从零实现 MHA

约 60 行核心代码,PyTorch,decoder-only 风格(带因果掩码)。先看骨架,再逐块讲解。

```python
import math
import torch
import torch.nn as nn
import torch.nn.functional as F


class MultiHeadAttention(nn.Module):
    """从零实现的多头自注意力(decoder-only,带因果掩码)。

    形状约定:B=batch, T=序列长, C=d_model, H=头数, D=C//H(每头维度)
    """

    def __init__(self, d_model: int, n_heads: int, dropout: float = 0.0):
        super().__init__()
        assert d_model % n_heads == 0, "d_model 必须能被 n_heads 整除"
        self.n_heads = n_heads
        self.head_dim = d_model // n_heads

        # 一次线性变换同时算出 Q、K、V(等价于三个独立矩阵,但只发射一次矩阵乘,GPU 更高效)
        self.qkv_proj = nn.Linear(d_model, 3 * d_model, bias=False)
        # 输出投影 W_o:融合各头信息
        self.out_proj = nn.Linear(d_model, d_model, bias=False)
        self.dropout = nn.Dropout(dropout)

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        B, T, C = x.shape

        # ① 戴帽子:一次算出 QKV,再切成三份
        qkv = self.qkv_proj(x)                      # (B, T, 3C)
        q, k, v = qkv.chunk(3, dim=-1)              # 各 (B, T, C)

        # ② 分头:C 维切成 H 份,把"头"换到 batch 一侧,让每个头独立做注意力
        #    (B, T, C) → (B, T, H, D) → (B, H, T, D)
        q = q.view(B, T, self.n_heads, self.head_dim).transpose(1, 2)
        k = k.view(B, T, self.n_heads, self.head_dim).transpose(1, 2)
        v = v.view(B, T, self.n_heads, self.head_dim).transpose(1, 2)

        # ③ 打分 + 缩放:注意除的是每头维度 D,不是 d_model!
        #    (B, H, T, D) @ (B, H, D, T) → (B, H, T, T)
        scores = (q @ k.transpose(-2, -1)) / math.sqrt(self.head_dim)

        # ④ 因果掩码:上三角(未来位置)填 -inf,softmax 后权重为 0
        causal_mask = torch.triu(
            torch.ones(T, T, dtype=torch.bool, device=x.device), diagonal=1
        )
        scores = scores.masked_fill(causal_mask, float("-inf"))

        # ⑤ 归一化成注意力权重,加权听取发言
        attn = F.softmax(scores, dim=-1)            # (B, H, T, T)
        attn = self.dropout(attn)
        out = attn @ v                              # (B, H, T, D)

        # ⑥ 合并各头 + 输出投影:transpose 后内存不连续,需 contiguous 才能 view
        out = out.transpose(1, 2).contiguous().view(B, T, C)
        return self.out_proj(out)
```

**逐块拆解,每一步都是考点:**

- **① 融合 QKV 投影**:三个 768×768 的矩阵乘合并成一个 768×2304——数学等价,但 GPU 上"一次大矩阵乘"远快于"三次小矩阵乘"(kernel 启动开销和访存,第 9 章细讲)。面试官看到你这么写,会知道你读过工业级实现。
- **② 那两行 view/transpose**:MHA 实现里最容易写错、也最常被要求"解释每一步形状"的地方。记住口诀:**切最后一维,把 H 换到前面去**。换到前面是为了让 `(T, D)` 成为矩阵乘的操作维度,H 和 B 一样被广播——h 个头的注意力一次批量算完,没有任何 for 循环。
- **③ 除以 √head_dim 而非 √d_model**:每个头在 D 维子空间里独立做点积,方差是 D 不是 C。写成 `math.sqrt(d_model)` 是经典错误,面试官专门盯这里。
- **④ mask 用 bool + masked_fill**:比乘法 mask 数值上更干净;`diagonal=1` 保证对角线(自己看自己)不被遮。
- **⑥ contiguous()**:transpose 只改视图不搬内存,view 要求内存连续——不加会直接报错。能主动解释这一句,说明你真跑过代码而不是背的。

### 2.3.4 验证:形状、因果性、与官方实现对拍

写完就测,三个测试分别对应三种"写错了自己却不知道"的 bug:

```python
torch.manual_seed(0)
B, T, C, H = 2, 16, 64, 4
mha = MultiHeadAttention(d_model=C, n_heads=H)
x = torch.randn(B, T, C)

# 测试 1:形状——输出必须和输入同形
out = mha(x)
assert out.shape == (B, T, C)

# 测试 2:因果性——改动"未来",不许影响"过去"
#   扰动第 10 个 token 之后的输入,前 10 个位置的输出必须一字不变
x2 = x.clone()
x2[:, 10:, :] = torch.randn(B, T - 10, C)
out2 = mha(x2)
assert torch.allclose(out[:, :10], out2[:, :10], atol=1e-6), "泄露未来信息!"

# 测试 3:与 PyTorch 官方 SDPA 对拍——同样的 Q/K/V,结果必须一致
q, k, v = mha.qkv_proj(x).chunk(3, dim=-1)
q = q.view(B, T, H, C // H).transpose(1, 2)
k = k.view(B, T, H, C // H).transpose(1, 2)
v = v.view(B, T, H, C // H).transpose(1, 2)
ref = F.scaled_dot_product_attention(q, k, v, is_causal=True)   # 官方融合实现
ref = ref.transpose(1, 2).contiguous().view(B, T, C)
ref = mha.out_proj(ref)
assert torch.allclose(out, ref, atol=1e-5), "与官方实现不一致!"

print("三项测试全部通过 ✓")
```

其中**测试 2 是精髓**:因果性 bug(比如 mask 的 diagonal 参数写错、mask 加在了 softmax 之后)不会报错、不会形状不对,模型照常训练甚至 loss 低得出奇——因为它在偷看答案。这类"静默作弊"只有靠这种行为测试才能抓住。把这个测试讲给面试官听,是展示工程素养的绝佳素材。

### 2.3.5 工程现实:你手写的这版,生产里没人用

必须知道的分寸感(面试加分点):手写 MHA 是用来**证明理解**的,生产代码里注意力这几行会被一句话替代:

```python
out = F.scaled_dot_product_attention(q, k, v, is_causal=True)
```

这个 SDPA 接口会自动分派到 FlashAttention 等融合 kernel:不显式生成 n×n 矩阵、不单独做 softmax,显存 O(n),速度快数倍。**数学完全等价,访存方式天壤之别**——为什么,第 10 章整章回答。手写版的价值在于:①面试白板;②理解 GQA/MLA 等变体时有参照;③给第 10 章"朴素实现慢在哪"提供 baseline。

## 2.4 ❓ 高频考题:一条真实的追问链

以下按真实面试的追问顺序排列。每题给"答案骨架"——面试时先给一句话结论,再按需展开。

**Q1:讲讲 self-attention。**
骨架:软查表/开会比方(30 秒)→ Q/K/V 来源(同一输入乘三个可学习矩阵)→ 公式 softmax(QKᵀ/√d)V → 一句点题:"输出是上下文相关的 token 表示"。忌:上来就背公式,不给直觉。

**Q2:为什么要除以 √d_k?**
骨架:点积方差随维度线性增长(可现场推:d 项独立、每项方差 1、和的方差为 d)→ softmax 输入过大会饱和 → one-hot 化 + 梯度消失 → 除以 √d 把方差归一。加分:能答上"为什么不是除以 d"。

**Q3:Q、K、V 为什么要用三个不同的投影矩阵?共用一个行不行?**
骨架:三种角色目标不同——匹配用的表示(K)和传递用的表示(V)各有各的最优方向,共用会互相牵制。极端例子:若 Q=K,则分数矩阵对称且对角线恒为最大(自己和自己点积最大),每个词都最关注自己,表达力大损。

**Q4:多头的作用?头数怎么选?**
骨架:单头只有一套注意力分布,多头让模型在不同子空间并行捕捉不同关系(指代/语法/语义);参数量与单头相同,是"划分"不是"加料"。头数:保持 d_head 在 64~128,h 随模型宽度等比增长。加分:提"事后可视化确实见到分工,但这是涌现不是设计"。

**Q5:attention 的计算和显存复杂度?瓶颈在哪?**
骨架:计算 O(n²d),显存 O(n²)(分数矩阵);长上下文时显存先爆。缓解:FlashAttention(不改数学改访存,O(n) 显存)、稀疏/线性注意力、以及推理侧的 KV cache 系列优化。这题答好,面试官大概率顺着问 KV cache——正中下怀,见第 13 章。

**Q6(手写题):白板写 MHA 的 forward,并说明每步张量形状。**
考的就是 2.3.3 的六步。高危陷阱:除以 √d_model(应为 √head_dim)、忘了 causal mask、忘了 contiguous、mask 填 0 而不是 -inf。写完主动说一句"我平时会加一个因果性单元测试"——直接把 Q7 递给自己。

**Q7:你怎么测试一个 attention 实现是对的?**
骨架:形状测试 → 因果性行为测试(扰动未来、检查过去不变)→ 与参考实现(SDPA)数值对拍。点题:因果性 bug 是静默的,loss 反而更好看,必须靠行为测试抓。
